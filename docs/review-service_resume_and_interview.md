# Review-Service 简历优化版 + 面试讲解手册

## 一、优化后简历描述（可直接替换）

评价服务为电商业务提供用户评价全链路能力，覆盖 C 端用户发布评价、O 端运营审核评价、B 端商家查看/回复评价等核心流程。系统围绕评价写入、审核流转、商家查询与回复等场景进行设计，支撑评价业务高并发读写与检索需求。

- 负责评价服务架构设计与演进，基于 CQRS 拆分读写模型：写侧落 MySQL，读侧基于 Elasticsearch 提供检索能力，并通过接口分层（service/biz/data）收敛业务规则与数据访问复杂度；
- 设计并实现 review-job，基于 Canal + Kafka 构建 MySQL->ES 异步同步链路，解耦主写链路与检索链路，保障检索侧可独立扩缩容；
- 持续优化读链路高可用：落地 Redis 双层缓存（hot/stale）+ singleflight 请求合并 + 服务端降级策略，在 ES 异常时可回退旧缓存结果，降低故障放大风险；
- 在查询热点接口引入按 `method + storeID` 维度限流隔离，避免单商家热点流量冲击全局，提升系统在突发流量场景下的稳定性；
- 实现 O 端 AI 审核能力接入，构建“超时控制 + 审核建议回写”流程，辅助运营识别风险评价并提升审核效率。

## 二、面试讲解（每条怎么讲）

> 建议讲法结构：**背景问题 -> 为什么做 -> 怎么做 -> 效果/收益 -> 复盘**。

### 1) CQRS 与读写拆分（MySQL + ES）

#### 为什么要做
- 评价写入和评价检索是两类不同负载：写链路强调事务一致性，读链路强调复杂检索与高吞吐。
- 如果所有查询都压在 MySQL，会出现索引膨胀、慢查询和扩展瓶颈。

#### 怎么实施
- 写侧以 MySQL 为真相源，承接创建评价、回复、申诉、审核流转。
- 读侧使用 ES 承接商家维度分页检索。
- 代码层按 `service -> biz -> data` 分层：service 负责协议转换，biz 管规则，data 管存储细节。

#### 面试可说的收益
- 读写职责边界清晰，后续优化能在读侧独立演进（缓存、限流、降级）而不影响主写链路。

#### 常见追问
- Q：为什么不直接全量 MySQL？
- A：检索场景对多字段过滤和分页吞吐要求高，ES 在这类场景更适合；MySQL 继续承担强一致写入真相源。

---

### 2) MySQL->ES 异步链路（Canal + Kafka + review-job）

#### 为什么要做
- 不能让主写接口同步写 ES，否则外部依赖抖动会直接拖垮核心交易链路。
- 需要把“写成功”与“可检索”解耦为最终一致。

#### 怎么实施
- 通过 Canal 订阅 MySQL binlog，写入 Kafka。
- review-job 消费变更事件并更新 ES 文档。
- review-service 仅负责写 MySQL 和查询 ES，不直接维护跨存储同步逻辑。

#### 面试可说的收益
- 主链路稳定性提升，检索链路可独立扩容；系统具备更好的故障隔离能力。

#### 常见追问
- Q：一致性延迟怎么处理？
- A：明确告知最终一致；读链路再叠加缓存策略与降级，降低短时延迟带来的用户体感波动。

---

### 3) 读链路高可用：hot/stale 缓存 + singleflight + 降级

#### 为什么要做
- 原方案只有单层缓存，ES 异常时容易放大故障。
- 高并发下同 key 回源会造成缓存击穿。

#### 怎么实施
- 设计两层 Redis：
  - `hot cache`（短 TTL）用于性能；
  - `stale cache`（长 TTL）用于兜底。
- 查询路径：
  1. 先读 hot；
  2. miss 回源 ES，成功后同时回填 hot/stale；
  3. ES 失败时读 stale 并返回降级结果；stale 也 miss 则返回 `DEPENDENCY_DEGRADED`。
- 使用 singleflight 合并相同 `storeID+offset+limit` 的并发请求，减少瞬时回源压力。
- 修复 singleflight 返回值不安全断言带来的潜在 panic 风险（先做类型校验）。

#### 面试可说的收益
- 在依赖异常场景下系统“可用优先”，不是直接雪崩失败。
- 并发回源被抑制，链路更平稳。

#### 常见追问
- Q：为什么允许返回旧数据？
- A：评价列表是弱一致读场景，可接受短时陈旧；优先保证服务可用性和响应稳定。

---

### 4) 热点隔离限流：`method + storeID`

#### 为什么要做
- 商家流量分布常呈长尾，热点商家可能在活动期间放大请求量。
- 全局限流会误伤普通商家，无法做到精细隔离。

#### 怎么实施
- 在 HTTP/GRPC 服务端中间件中，仅对 `ListReviewByStoreID` 接口生效。
- 限流 key 采用 `operation + storeID`，为每个 key 维护独立 token bucket。
- 超限统一返回 `RATE_LIMITED`，便于网关和客户端识别与治理。

#### 面试可说的收益
- 保障全局稳定性的同时，控制热点局部影响范围。

#### 常见追问
- Q：为什么不按用户限流？
- A：这个接口的主要压力来源是商家查询维度，按 storeID 隔离更贴近真实流量形态。

---

### 5) 服务治理与可观测性增强

#### 为什么要做
- 高可用策略如果没有观测信号，很难排障，也无法评估治理效果。

#### 怎么实施
- 新增语义化错误码：`RATE_LIMITED`、`DEPENDENCY_DEGRADED`。
- 在响应头输出：
  - `X-Review-Degraded`：是否走降级；
  - `X-Review-Cache-Layer`：数据来源（hot/stale/es）。
- 结构化日志增加 `store_id/offset/limit/cache_layer/degraded` 字段。

#### 面试可说的收益
- 能快速定位“为什么这次慢/为什么这次是旧数据/为什么被限流”。
- 为后续 SLO 指标建设（降级率、命中率、限流命中率）打基础。

---

### 6) 审核链路细节：状态机一致性与 AI 审核

#### 为什么要做
- 状态机入口不一致会引发流程错误（例如创建后状态不在待审核）。

#### 怎么实施
- 将创建评价默认状态统一为待审核（`10`），与审核规则对齐。
- AI 审核链路保留超时边界和结果回写逻辑，保障审核流程可控。

#### 面试可说的收益
- 避免状态漂移导致的业务异常，提升审核流程稳定性。

## 三、1 分钟项目话术（可直接背）

我负责的是电商评价中台服务，核心做了三件事：第一是用 CQRS 拆分读写，写侧 MySQL 保证强一致，读侧 ES 承担检索；第二是通过 Canal+Kafka+review-job 做异步同步，把主写链路和检索链路解耦；第三是重点做了读链路高可用，落地了 hot/stale 双层缓存、singleflight 请求合并和降级策略，并在热点查询接口加了 method+storeID 维度限流。这样在 ES 抖动或热点流量冲击时，服务仍能稳定返回，同时通过错误码和响应头输出降级与缓存层信息，排障和治理闭环也更清晰。

## 四、面试官继续深挖时的回答模板

- 如果问“你的优化有数据吗？”
  - 可以回答：我重点跟踪了缓存命中率、降级命中率、限流命中率、接口 P95/P99，目标是把依赖抖动时的错误率和超时率压下去。
- 如果问“最大的取舍是什么？”
  - 可以回答：读链路选择了可用优先，允许短时陈旧；写链路仍坚持 MySQL 真相源和流程状态一致性。
- 如果问“下一步会做什么？”
  - 可以回答：补充熔断器和更细粒度指标看板，把降级策略从静态配置升级为动态开关。
